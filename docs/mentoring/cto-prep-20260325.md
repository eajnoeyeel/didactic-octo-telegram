# 서론

## 연구 필요성

MCP 생태계가 빠르게 확장되면서, LLM이 사용할 수 있는 외부 Tool 수는 계속 늘어나고 있습니다. 문제는 Tool이 많아질수록 LLM이 더 똑똑해지는 것이 아니라, 오히려 **무엇을 써야 하는지 결정하는 비용과 난이도**가 급격히 커진다는 점입니다.

특히 MCP 클라이언트는 연결된 서버의 `tools/list` 결과를 바탕으로 동작하므로, 실무에서는 아래 두 문제가 동시에 발생합니다.

- **Prompt bloat**: 연결된 서버 수가 많아질수록 Tool description이 컨텍스트를 잠식함
- **Selection complexity**: 유사한 Tool이 많아질수록 LLM의 선택 오류 가능성이 증가함

예를 들어:

```text
서버 10개 × 서버당 도구 20개 × 도구 정의 300토큰 = 60,000토큰
```

이 문제는 단순히 "더 좋은 프롬프트를 쓰자"로 해결되지 않습니다.  
핵심은 **도구 사용(tool use)** 이전에 **도구 탐색(tool discovery)** 을 어떻게 외부화할 것인가입니다.

이 프로젝트는 바로 그 지점에 답하려고 합니다.

- **LLM 쪽 가치**: 우리 MCP 하나만 연결하면 필요한 Tool을 검색하고 실행할 수 있음
- **Provider 쪽 가치**: 내 Tool이 왜 선택되지 않았는지 analytics로 보고 description 개선 가이드를 받을 수 있음

## 선행 연구

이 프로젝트의 방향은 최근 MCP/tool-use 연구 흐름 위에 놓여 있습니다.

| 논문 | 핵심 결과 | 프로젝트 시사점 |
| --- | --- | --- |
| [RAG-MCP (Gan & Sun, 2025)](https://arxiv.org/abs/2505.03275) · [정리](../papers/rag-mcp-analysis-ko.md) | prompt token을 크게 줄이고 tool selection accuracy를 **43.13% vs 13.62%**로 개선 | discovery를 LLM 바깥 retrieval layer로 분리하는 접근의 타당성 |
| [MCP Tool Descriptions Are Smelly! (Hasan et al., 2026)](https://arxiv.org/abs/2602.14878) · [정리](../papers/mcp-descriptions-smelly-analysis-ko.md) | 103개 서버, 856개 도구 중 **97.1%**가 최소 1개 smell 보유, augmentation 후 task success **+5.85pp** | description quality scoring과 개선 가이드의 필요성 |
| [From Docs to Descriptions (Wang et al., 2026)](https://arxiv.org/abs/2602.18914) · [정리](../papers/from-docs-to-descriptions-analysis-ko.md) | 10,831개 MCP 서버 분석, compliant description의 selection probability가 **72% vs 20%** | description이 cosmetic field가 아니라 ranking signal이라는 근거 |
| [Tool Preferences in Agentic LLMs are Unreliable (Faghih et al., EMNLP 2025)](https://aclanthology.org/2025.emnlp-main.1060/) · [정리](../papers/tool-preferences-unreliable-analysis-ko.md) | 설명 편집만으로 일부 모델에서 **10배 이상 더 많이 선택**됨 | description A/B 실험과 fairness guardrail의 필요성 |
| [Quality Matters (Iskander et al., 2024)](https://arxiv.org/abs/2409.16341) · [정리](../papers/quality-matters-analysis-ko.md) | 검증 없는 synthetic-only 데이터보다 quality-checked 데이터가 더 신뢰할 만함 | seed set + synthetic + quality gate 조합의 근거 |

요약하면 선행 연구는 세 가지를 말합니다.

1. **retrieval layer는 필요하다**
2. **description은 실제 선택을 바꾼다**
3. **evaluation은 synthetic-only로 가면 위험하다**

## 연구 목적과 가설

이 연구의 목적은 단순히 "MCP Tool을 잘 찾는 시스템"을 만드는 데 그치지 않습니다. 더 중요한 목적은 **어떤 설계가 실제로 잘 작동하는지**, 그리고 **description 품질이 selection quality를 어떻게 바꾸는지**를 실험으로 보이는 것입니다.

핵심 가설은 아래와 같습니다.

> **Description 품질이 높을수록 Tool 선택률이 높아진다.**

이를 위해 아래 세 가지를 검증하려 합니다.

| 목표 | 연구 질문 | 기대 효과 |
| --- | --- | --- |
| Retrieval architecture 검증 | 1-Layer보다 2-Layer가 실제로 더 나은가 | Bridge 구조의 기술적 정당화 |
| Description 가설 검증 | description 품질이 selection rate를 바꾸는가 | Provider analytics의 제품 가치 증명 |
| 실험 프레임 구축 | 전략/임베딩/reranker를 같은 하네스로 비교할 수 있는가 | 향후 반복 실험 기반 확보 |

핵심 가설 검증은 **3중 증거 구조**로 설계합니다.

| 증거 유형 | 질문 | 방법 | 성공 기준 |
| --- | --- | --- | --- |
| 인과적 | 설명을 바꾸면 선택률이 실제로 바뀌는가 | Poor vs Good description A/B | Lift > 30%, p < 0.05 |
| 상관적 | quality와 selection이 전체 pool에서도 같이 움직이는가 | Spearman 상관 | r > 0.6, p < 0.05 |
| 설명적 | 어떤 quality 요소가 가장 중요한가 | OLS / feature importance | R² > 0.4 |

# 이론적 배경

## 연구 배경

이 프로젝트의 기술적 핵심은 **Bridge/Proxy + Retrieval** 조합입니다.

```text
LLM Client
  -> 우리 MCP 1개만 연결

Bridge MCP Server
  -> find_best_tool(query)
  -> execute_tool(tool_id, params)

Retrieval Layer
  -> embedding search / reranking / confidence branch

Provider MCP Servers
  -> 실제 Tool 실행
```

LLM에 노출되는 인터페이스는 최소화합니다.

| 노출 Tool | 역할 |
| --- | --- |
| `find_best_tool(query)` | 질의에 맞는 Tool 후보 추천 |
| `execute_tool(tool_id, params)` | 선택된 Tool을 실제 Provider MCP로 실행 |

이론적으로는 다음 세 층이 중요합니다.

1. **Externalized discovery**  
   Tool 전체를 프롬프트에 넣지 않고, discovery를 외부 검색 계층으로 분리

2. **Two-stage retrieval**  
   임베딩 검색으로 후보를 줄이고, reranker로 재정렬

3. **Confidence-aware serving**  
   확신이 높으면 Top-1, 낮으면 Top-K와 disambiguation 힌트를 제공

즉, 이 연구는 MCP를 "연결 문제"가 아니라 **탐색, 랭킹, 라우팅 문제**로 바라봅니다.

## 경쟁 환경

현재 시장에서 가까운 비교 대상으로는 **Smithery**와 **MetaMCP**를 볼 수 있습니다.

| 구분 | 현재 제공 가치 | 확인된 한계 | 우리 프로젝트와의 차이 |
| --- | --- | --- | --- |
| [Smithery](https://smithery.ai/) · [List all servers API](https://smithery.ai/docs/api-reference/servers/list-all-servers) | MCP server registry, 서버 검색 API, 배포/연결 진입점 제공 | 공식 문서 기준 `q` 파라미터로 semantic/full-text 검색은 가능하지만, **사용자 쿼리 -> 최적 서버/도구 자동 라우팅** 기능은 명시되지 않음 | 우리는 registry가 아니라 query-native retrieval과 selection을 다룸 |
| [MetaMCP](https://metamcp.com/) · [Docs](https://docs.metamcp.com/en) · [GitHub](https://github.com/metatool-ai/metamcp) | 여러 MCP 서버를 하나의 unified MCP server로 aggregation, namespace, middleware, endpoint hosting 지원 | 공식 문서/README는 aggregation과 middleware에 강점이 있고, tool selection은 `coming soon`으로 표현됨 | 우리는 aggregation 자체보다 **어떤 query에 어떤 tool을 보여줄지**와 그 품질 측정을 다룸 |

Smithery에 대한 제 해석은 이렇습니다.

- 현재 문서상 가능한 것은 **registry 검색 API를 이용해 개발자가 워크플로우를 직접 짜는 방식**입니다.
- 실무 대안으로는 **여러 서버를 미리 연결해 두고 LLM이 노출된 tool 목록에서 고르게 하는 방식**이 있습니다.
- 즉, **"쿼리 -> 서버/도구 자동 라우팅"은 아직 플랫폼 기능이라기보다 개발자가 직접 구현해야 하는 영역**으로 보입니다.

MetaMCP에 대한 제 해석은 이렇습니다.

- MetaMCP는 **aggregation, gateway, middleware**에는 강합니다.
- 하지만 그것만으로는 **prompt bloat, query-aware ranking, description quality analytics** 문제가 해결되지 않습니다.
- 다시 말해 MetaMCP는 "여러 서버를 하나로 묶는 인프라"이고, 이 프로젝트는 그 위에서 필요한 **retrieval intelligence layer**를 정의하려는 시도입니다.

## 차별성

이 주제가 가지는 차별성은 세 가지입니다.

1. **Registry가 아니라 retrieval을 다룬다**  
   "어떤 MCP가 존재하는가"가 아니라 "현재 사용자 질의에 무엇을 노출해야 하는가"를 다룸

2. **Aggregation이 아니라 selection quality를 다룬다**  
   여러 서버를 합치는 것만으로는 끝나지 않으며, 랭킹과 오선택 감소가 핵심임

3. **추천만이 아니라 analytics loop를 다룬다**  
   description 품질 -> 선택률 -> 개선 가이드의 피드백 루프를 제품화 대상으로 삼음

즉, 이 프로젝트는 **MCP 연결 인프라**와 **MCP 추천 인텔리전스** 사이의 빈 공간을 메우려는 시도입니다.

# 연구 방법

## 설계 원칙

이번 단계에서 제 원칙은 복잡한 기능을 빨리 많이 붙이는 것이 아니라, **가장 큰 가정부터 순서대로 검증하는 것**입니다.

제가 잡은 기준은 네 가지입니다.

1. **먼저 baseline을 만든다**  
   지금은 `flat`과 `sequential` 두 가지가 이미 있으므로, 이 둘을 먼저 비교할 수 있습니다.

2. **한 번에 변수 하나만 바꾼다**  
   전략, 임베딩, reranker, description 품질을 한꺼번에 바꾸면 왜 결과가 좋아졌는지 알 수 없습니다.

3. **구조가 맞는지 먼저 보고, 그다음 정교화한다**  
   2-Layer 자체가 의미 없으면 그 위에 reranker나 taxonomy를 쌓는 것은 우선순위가 떨어집니다.

4. **평가 데이터는 적더라도 믿을 수 있어야 한다**  
   synthetic 데이터를 많이 만드는 것보다, 수동 검증한 seed set과 quality gate가 더 중요합니다.

## 실험 순서

실험은 아래 순서로 진행합니다.

| 단계 | 지금 확인할 것 | 왜 먼저 하는가 |
| --- | --- | --- |
| **E0** | 1-Layer가 나은지, 2-Layer가 나은지 | 전체 아키텍처 방향을 결정하는 가장 큰 가정이기 때문 |
| **E1** | 2-Layer를 쓴다면 sequential이 좋은지, parallel이 좋은지 | E0가 끝나야 비교 의미가 생김 |
| **E2** | 어떤 임베딩이 더 잘 찾는지 | 구조가 정해진 뒤에 모델 선택을 해야 낭비가 적음 |
| **E3** | reranker를 붙였을 때 실제 이득이 있는지 | 검색 구조가 잡힌 뒤에 붙여야 효과를 분리해 볼 수 있음 |
| **E4** | description을 바꾸면 selection rate가 실제로 바뀌는지 | retrieval 기반이 안정화된 뒤에야 핵심 가설을 깔끔하게 검증할 수 있음 |

여기서 가장 중요한 실험은 **E0**입니다.

- **1-Layer**: 전체 Tool을 한 번에 검색
- **2-Layer**: 먼저 서버를 고르고, 그 서버 안에서 Tool을 다시 검색

2-Layer는 검색 범위를 줄일 수 있다는 장점이 있지만, 첫 단계에서 서버를 잘못 고르면 정답 Tool이 아예 후보에서 사라질 수 있습니다.  
그래서 E0는 단순 비교가 아니라, **"서버를 먼저 고르는 구조가 실제로 이득인가?"** 를 확인하는 실험입니다.

제 판단은 이렇습니다.

- E0에서 2-Layer가 이득이면: server-first 구조, reranker, server analytics에 계속 투자
- E0에서 2-Layer가 별 이득이 없으면: flat 또는 parallel 중심으로 다시 정리

즉, **E0 결과가 이후 설계 우선순위를 정합니다.**

## 판단 기준

이번 단계에서는 지표도 많이 보기보다, 의사결정에 직접 필요한 것만 봅니다.

| 볼 것 | 이유 |
| --- | --- |
| **Precision@1** | 최종 추천이 맞았는지 보는 가장 중요한 숫자 |
| **Recall@K** | 정답 Tool이 후보군에 올라오기는 하는지 확인 |
| **Latency** | 좋아도 너무 느리면 실제 시스템으로 쓰기 어려움 |
| **실패 원인 분해** | 서버 단계에서 놓쳤는지, Tool 단계에서 밀렸는지 구분해야 다음 액션이 정해짐 |

Ground truth는 아래처럼 가져갑니다.

- **seed set**: 사람이 직접 확인한 정답 세트
- **synthetic set**: LLM이 만들되 quality gate를 통과한 것만 사용
- **원칙**: synthetic-only 결과를 그대로 믿지 않음

North Star는 당장은 하나만 봅니다.

- **Precision@1 >= 50%** at Pool 50, mixed domain

이 수치가 안정적으로 나오지 않으면, 그 다음 단계 최적화보다 먼저 구조 자체를 다시 봐야 합니다.
