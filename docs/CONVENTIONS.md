# 문서 관리 규약

> 최종 업데이트: 2026-03-22
> 목적: papers/와 research/ 문서의 작성 템플릿, 교차 참조 규칙, 네이밍 규약 정의

---

## 1. papers/ 템플릿

papers/ 폴더의 각 파일은 **개별 논문 아카이브**다. 논문 하나당 파일 하나가 대응된다.

모든 papers/ 파일은 아래 구조를 따른다:

### 필수 섹션

| 섹션 | 설명 |
|------|------|
| **논문명** | 원제 (영문) |
| **출처/링크** | arXiv ID, 학회명, DOI 등 |
| **한 줄 요약** | 이 논문의 핵심 기여를 1문장으로 |
| **해결하려는 문제** | 논문이 다루는 연구 문제 |
| **핵심 아이디어** | 제안하는 방법론의 핵심 |
| **방법론** | 구체적 구현/실험 방법 |
| **주요 결과** | 핵심 수치 포함 |
| **장점** | 기존 방법 대비 강점 |
| **한계** | 저자가 인정하거나 우리가 식별한 한계 |
| **프로젝트 시사점** | MCP Discovery Platform 관점에서의 의미 |
| **적용 포인트** | 우리 시스템의 어디에 반영하는가 |
| **관련 research 문서** | 이 논문을 참조하는 `../research/*.md` 링크 리스트 |

---

## 2. research/ 템플릿

research/ 폴더의 각 파일은 **특정 기능/문제/지표 해결 단위의 조사 문서**다. "무엇을 만들거나 개선하기 위해 어떤 논문/자료를 참고했는가"에 답하는 문서다.

모든 research/ 파일은 아래 구조를 따른다:

### 필수 섹션

| 섹션 | 설명 |
|------|------|
| **조사 목적** | 무엇을 결정/해결하기 위한 조사인가 |
| **해결하려는 기능/문제** | 구체적 기능명 또는 DP 번호 |
| **검토한 논문/자료 목록** | `../papers/*.md` 링크 리스트 |
| **각 자료에서 가져온 핵심 포인트** | 논문별 1-3줄 요약 |
| **후보 접근 방식 비교** | 표 형태 권장 |
| **채택안 / 제외안** | 최종 결정과 제외된 대안 |
| **판단 근거** | 왜 이 방향을 선택했는가 |
| **프로젝트 반영 방식** | 어떤 코드/설계에 어떻게 반영하는가 |
| **관련 papers** | 참조한 `../papers/*.md` 링크 리스트 (양방향 확인용) |

## 3. 교차 참조 규칙

### 원칙: 양방향 추적 가능

- **papers/ → research/**: 각 papers/ 파일의 "관련 research 문서" 섹션에, 이 논문을 참조하는 research/ 문서를 나열한다.
- **research/ → papers/**: 각 research/ 파일의 "검토한 논문/자료 목록" 섹션에, 참조한 papers/ 파일을 나열한다.

### 추적 가능해야 하는 질문

이 체계로 아래 질문에 답할 수 있어야 한다:

1. "이 기능을 만들 때 어떤 논문을 참고했지?" → research/ 문서의 "검토한 논문 목록" 확인
2. "이 논문은 어떤 기능 검토 과정에서 참고했지?" → papers/ 문서의 "관련 research 문서" 확인
3. "이 research 문서의 근거가 되는 개별 논문은?" → research/ 문서의 "관련 papers" 확인
4. "이 논문은 단독 참고였나, 여러 논문 비교의 일부였나?" → papers/ 문서의 "관련 research 문서" 개수로 판단

### 업데이트 절차

1. **새 paper 추가 시**: papers/ 파일 작성 → 해당 paper를 참조하는 research/ 문서에 링크 추가
2. **새 research 추가 시**: research/ 파일 작성 → 참조 papers/ 각각에 역방향 링크 추가
3. **기존 문서 변경 시**: 해당 문서의 "최종 업데이트" 날짜 갱신 + 교차 참조 양방향 확인

## 5. 폴더 역할 요약

| 폴더 | 역할 | 단위 |
|------|------|------|
| `context/` | 프로젝트 개요, 목표, 타임라인 | 프로젝트 전체 |
| `design/` | 아키텍처, 지표, 실험, GT, 코드 구조 | 설계 결정 단위 |
| `plan/` | 구현 계획, 후순위, 체크리스트 | 실행 단위 |
| `mentoring/` | CTO 질문, 미결 사항 | 의사결정 단위 |
| `research/` | 기능/문제별 조사 (다수 논문 묶음) | 조사 목적 단위 |
| `papers/` | 개별 논문 아카이브 | 논문 1개 단위 |

## 6. 작성 스타일 (Claude Code 지침 기준)

- **파일당 200줄 이하** — 초과 시 분할 후 상호 참조 추가
- 마크다운 헤더(`#`, `##`, `###`) + 글머리 기호로 구조화 — 밀집된 산문 지양
- 구체적·검증 가능한 지침: `"BGE-M3 사용"` (O), `"적절한 모델 선택"` (X)
- **결정 → 근거** 순서로 기술 — 서사적 설명·배경 나열 지양
- 분할 시 상호 참조 필수: `"상세: ./filename.md"` 형태로 포인터 추가
- 테이블은 `| col |` 형식 사용, 3열 이상 비교 시 표 권장
- 날짜 기입 필수: 파일 상단 `> 최종 업데이트: YYYY-MM-DD`
- 임의의 수치는 "초기값"으로 명시하고 calibration 계획 기재
